## §A — casual/tightened (recommended)

> we analyze modules more independently, so the memory issues at push
> time aren't as big of a deal

Yeah, saw that — helped most people, and we're not hitting it at
doxy.me's current scale. But there's a codec-library-specific case it
can't reach, and I think I've got it solved. Curious for your read.

The gist: zodvex lets codecs (bidirectional transforms — what powers our
SensitiveField encryption) live in the schema, so they run at every
read/write. For that to be automatic, the DB reader/writer need a central
lookup for "which fields on which table are codecs." Today that's a
codegen registry statically imported into every endpoint bundle — and
with no dynamic `import()` in the runtime, there's no way to lazy-load one
table's schema on first touch. Under zod v4 each model's ~0.2–0.3 MB at
eval, so every per-entrypoint isolate re-pays the whole graph. Naively
that OOMs a codec app around 100–150 tables.

The fix I just shipped: codegen a minimal per-table "descriptor" carrying
only the codec fields (everything else passes through) instead of the
full schema. Same apps now deploy clean at 750 tables and hit your
TooManyReads wall at 800 — right alongside a pure-`defineTable` baseline.
So codecs are at parity with plain Convex. Still need to bring it into the
real doxy.me project to confirm, but the numbers look solid.

Two things I'd genuinely value your take on — bigger one first:
1. My measurement harness (link). Every cell's a real `convex dev --once`
   with resets + runtime smoke checks. I'm targeting table-count parity
   with raw convex-values, but I don't have a feel for where the common
   ceiling is across other libraries, or whether that's even the right
   thing to be measuring.
2. I'd honestly rather be less clever than this. Two runtime primitives
   would let me delete most of it: deterministic lazy module loading in
   the default runtime (content's fixed per deploy, so a cached
   `loadModule()` stays deterministic), or evaluating identical `_deps`
   chunks once per deployment instead of per entrypoint. No urgency —
   just flagging a codec library as a ready consumer if either's ever on
   the roadmap.

## §B — detailed/original

Hey Ian — following up on the memory thread with something concrete. You
mentioned per-entrypoint analysis mostly retired the push-time memory
issues, and for plain apps that's been true for us too. But building
zodvex surfaced a case it structurally can't help, and I'd love your read
on whether there's a platform-side primitive that could.

**The shape of the problem:** a codec library (we auto-encode/decode
fields like our `SensitiveField` at the `ctx.db` boundary) needs
per-table schema info available wherever `ctx.db` runs. Since `db.get(id)`
can reach any table (relational lookups), that knowledge has to be global
to the isolate — and with dynamic `import()` unavailable in the default
runtime, "global" means one module statically importing every model.
Zod v4 evaluation costs ~0.2–0.3 MB per model, and per-entrypoint analysis
multiplies that by every endpoint. Measured on real deploys (fresh-diff,
reset between cells): the codec-on shape OOMs at ~100–150 models, while
the identical app with the central graph removed deploys clean at 800 —
right at the same TooManyReads wall as a pure-`defineTable` baseline we
ran the same day. So per-entrypoint analysis fixed apps whose weight is
spread across files; it can't help a library whose weight is one shared
registry every entrypoint must re-evaluate.

We also verified the obvious library-side fix (each endpoint registers
only the models it imports) and it OOM-proofs everything. Its catch is
relational lookups — an endpoint that never imported the looked-up
table's model can't decode it; in the naive form that's a silent miss,
though enforced `db.get(table, id)` + a tiny manifest turn it into a loud
throw. We set it aside not for that but because it reaches the same
TooManyReads ceiling as the minimal-central-map approach while costing
import discipline (or a build step) and a consumer-migration dependency.
The underlying constraint is the runtime's, not the analyzer's.

**Two primitives that would dissolve it entirely (either one):**

1. **Deterministic lazy module loading in the default runtime** — full
   dynamic `import()`, or even a narrower `loadModule(specifier)` that's
   content-addressed within the deployment version (so determinism is
   preserved — module content is immutable per deploy) and cached after
   first load. Our wrapper would lazy-load one table's schema on first
   access; both analysis and runtime memory become proportional to tables
   actually touched.
2. **Shared evaluation of split chunks across entrypoint isolates** — the
   registry chunk is byte-identical in every endpoint's bundle; evaluating
   it once per deployment instead of once per entrypoint removes the
   multiplier without new JS semantics.

(A third, smaller one: an official hook for build-time manifests/bundler
plugins would let tooling like ours solve more of this ourselves.)

Happy to share the full measurement methodology — it's all in the repo
(`examples/stress-test/results/`, every number is a real `convex dev
--once` push with deployment resets and runtime smoke checks). Is either
primitive plausible on your side, or is there a constraint I'm not seeing?

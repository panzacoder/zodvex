# Spike: convex-only schema, lazy zod — feasibility & shape

Goal: at deploy time, can the schema-eval isolate import a pure-Convex
artifact (no zod, no zodvex) while keeping the runtime path zod-flavored?

Verdict: **Yes, mechanically.** The codegen step that already
dynamic-imports models can also emit a Convex-only `_zodvex/tables.ts`.
Round-trip from live `TableDefinition` to source is straightforward
because every needed field is public. No private API needed.

## What's exposed on a Convex `TableDefinition`

`node_modules/convex/dist/esm/server/schema.js`:

```
class TableDefinition {
  validator            // VObject (the document validator)
  indexes              // [{ indexDescriptor, fields }]
  stagedDbIndexes
  searchIndexes        // [{ indexDescriptor, searchField, filterFields }]
  stagedSearchIndexes
  vectorIndexes        // [{ indexDescriptor, vectorField, dimensions, filterFields }]
  stagedVectorIndexes
  export()             // public; returns the JSON the backend consumes
}
```

Validators (`VString`, `VObject`, `VId`, etc., in
`convex/dist/esm/values/validators.js`) each expose a public
`kind` discriminator plus per-kind public fields (`tableName`,
`fields`, `members`, etc.) and `isOptional: "required" | "optional"`.

## Round-trip prototype (actual output, real `TaskModel`)

```ts
export const TaskTable = defineTable({
  "title": v.string(),
  "description": v.optional(v.string()),
  "status": v.union(v.literal("todo"), v.literal("in_progress"), v.literal("done")),
  "priority": v.union(v.union(v.literal("low"), v.literal("medium"), v.literal("high")), v.null()),
  "ownerId": v.string(),         // ← zodvex's zx.id() maps to v.string today (pre-existing)
  "assigneeId": v.optional(v.string()),
  "dueDate": v.optional(v.float64()),
  "completedAt": v.optional(v.float64()),
  "estimate": v.optional(v.float64()),
  "createdAt": v.float64()
})
  .index("by_owner", ["ownerId"])
  .index("by_status", ["status"])
  .index("by_assignee", ["assigneeId"])
  .index("by_created", ["createdAt"])
  .index("by_completed", ["completedAt"])
```

Generated by a ~30-line walker over `validator.kind` + arrays. The
"ownerId is `v.string`" note is faithful — that's what zodvex's
existing mapping produces. If `zx.id` ever upgrades to `v.id(...)`,
the round-trip picks it up automatically.

## How it slots in

The cleanest shape preserves Convex's expected workflow:

```ts
// convex/schema.ts — user-authored, still owns the schema (Convex-canonical)
import { defineSchema } from 'convex/server'
import { TaskTable, UserTable } from './_zodvex/tables'

export default defineSchema({
  tasks: TaskTable,
  users: UserTable,
})
```

User still calls `defineSchema(...)`. User still owns `convex/schema.ts`.
Only the validators come from a generated file, the way other Convex
generated files already do (`_generated/api`, `_generated/server`).

`_zodvex/tables.ts` is small (~one `defineTable({...}).index(...)` chain
per model). Pure `convex/server` + `convex/values` imports.

## Runtime side

`initZodvex` today builds `__zodTableMap` inside `defineZodSchema`. Under
this proposal, schema.ts is plain Convex — no `__zodTableMap` is built
during schema-eval. Options for the runtime:

- **A.** Emit `_zodvex/tableMap.lazy.ts` (mirrors `api.lazy.ts`). Pass
  it to `initZodvex` as `tableMap: zodTableMap`. Lazy thunk dynamic-imports
  the heavy zod schemas on first DB call. Endpoint files keep their
  existing `import { TaskModel } from './models/task'` for zq/zm —
  unchanged.
- **B.** `initZodvex` resolves the table map on first use by dynamic-
  importing user model files itself. More implicit; less explicit
  about source of truth. Probably worse.

I'd go with A. It mirrors the api.lazy pattern we just shipped.

## Estimated headroom

Cross-library data from yesterday says pure-Convex schema-eval handles
**≥ 2000 endpoints** before hitting the TooManyReads limit at
finish_push (a separate non-memory limit). The schema-eval ceiling
for zodvex apps under this proposal would jump from ~155 (today) or
~700 (zodvex-mini + slim) to **~2000**, putting it on par with plain
convex apps. **~13× over today's default, ~3× over today's best
configuration.**

## What this is NOT

- Not the #63 compile-away. Endpoints still load zod at runtime.
- Not a source rewriter. Codegen reads the live `TableDefinition`
  object — same mechanism used today for the registry.
- Not a userland API change beyond the one import path swap in
  `schema.ts` (TaskModel → TaskTable from generated tables file).

## Convex-alignment concerns (your flag)

The proposal stays close to Convex's expected pattern in three ways:

1. **User still authors `convex/schema.ts`.** It's their file, with
   `defineSchema({...})` and table references — exactly what Convex
   docs show.
2. **The generated `_zodvex/tables.ts` parallels `_generated/api.ts`.**
   Convex already trains users to import generated artifacts from
   inside `convex/`. We're adding a new sibling generated file, not
   subverting the model.
3. **No magic.** The validators emitted are plain Convex `v.*` calls;
   the schema reads like hand-authored Convex. A Convex dev opening
   `_zodvex/tables.ts` sees a recognizable Convex schema.

One thing that would push us outside Convex's comfort zone:
auto-generating `convex/schema.ts` itself. Don't do that. Keep
schema.ts user-authored. Generate only the table validators.

## Risks worth tracking

1. **Source-of-truth divergence.** If the user edits a model field
   and forgets to run `zodvex generate` before `convex deploy`, the
   deployed schema is stale. Same risk as the registry today, but
   the surface is larger (schema vs. runtime). Mitigation: make
   `zodvex generate` mandatory before deploy in docs; consider a
   `zodvex check` that fails CI if any generated file is stale.
2. **Validator coverage.** The spike's walker handles the common
   kinds (string/number/boolean/literal/array/object/union/record/
   null/any/id/int64/bytes/optional). Any custom validator zodvex
   maps to (e.g., new Convex 1.4+ types) needs a walker case. Easy
   to extend incrementally.
3. **`zodToConvex` deferred semantics.** If zodvex's mapping ever
   produces validators that depend on dynamic runtime info, the
   round-trip would freeze that info at codegen time. Today's mapping
   is purely structural, so this isn't a concern, but worth flagging.

## What I'd want to confirm before implementing

- That `defineSchema({ ...tables })` accepts `TableDefinition` objects
  imported from a sibling file (vs requiring inline construction).
  Easy to test; expect yes since defineSchema just iterates entries.
- That the round-trip on a `.searchIndex(...)` and `.vectorIndex(...)`
  actually preserves them on a real model that uses them. The spike
  only tested basic indexes.
- That the convex codegen step (the `_generated/` files) runs
  AFTER zodvex generate. Otherwise the typed `DataModel` won't know
  about tables from a freshly regenerated `_zodvex/tables.ts`.

## Estimated scope

If we proceed:
- Codegen change: new `generateTablesFile()` function (~80 lines).
  Walker: ~30 lines. Wiring into `commands.ts:generate()`: ~5 lines.
- Library change: add `tableMap?` option to `initZodvex` for the lazy
  loader; small.
- Codegen: emit `_zodvex/tableMap.lazy.ts` analogous to `api.lazy.ts`.
  ~10 lines.
- Migrate transform: rewrite `defineZodSchema({...})` calls in user
  schema.ts to `defineSchema({...})` with `TableX` from `_zodvex/tables`.
  Coordinated edits but mechanical.
- Examples + docs: update task-manager and quickstart to the new shape.

Smaller surface than the api.lazy work we just shipped. No new
dependencies, no AST tools.

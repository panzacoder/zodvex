# Per-endpoint baseline — 2026-05-12

First measurement under the new Convex backend model: each entrypoint is bundled and
analyzed independently. The harness in this repo (`bench.ts`) mirrors that:
esbuild per entry with Convex's exact isolate config, then a fresh `node --expose-gc`
subprocess per bundle that reports heap delta on load.

Seeds: 7 hand-written models × 5 flavors, fanned to N copies. Each endpoint imports
only its own model — there is no cross-table fan-in in this corpus.

## Numbers at N=200, sample=40

| Flavor | Bundle bytes (max) | Heap delta MB (max) | RSS MB (max) |
|---|---:|---:|---:|
| `convex` (plain) | 59 KB | 0.41 | 32.8 |
| `convex-helpers-zod3` | 155 KB | 1.21 | 39.5 |
| `convex-helpers` (zod4) | 471 KB | 2.29 | 55.8 |
| `zodvex` (full zod) | 516 KB | 2.70 | 59.0 |
| `zodvex-mini` | 541 KB | 2.95 | 61.4 |

All measurements stable across endpoints (min ≈ max within each flavor) and
unchanged between N=5 and N=200 — confirming the harness measures truly
per-entry cost, not aggregate.

## Headroom

The per-endpoint ceiling under Convex's new backend behavior is roughly the
isolate's 64 MB heap budget. With these seeds:
- zodvex sits at **~4% of the budget per endpoint** (2.7 MB / 64 MB).
- Worst flavor is `zodvex-mini` at **4.6%**.

For these seeds we have ~20× the headroom per endpoint. The new constraint is
not currently the binding one.

## What this corpus does NOT measure

These seeds have no fan-in: each endpoint imports a single model and its own
function builders. Real apps differ in three ways that compound a single
endpoint's cost:

1. **Registry import.** zodvex's codegen emits `_zodvex/api.ts` referencing
   every schema. If `za`-style actions import it for auto-decode, every such
   endpoint inherits the full graph.
2. **Cross-table reads.** `ctx.db.get('otherTable')` is typed against the
   other table's model — the endpoint bundle pulls that model in.
3. **Schema complexity.** Our seeds use ~4-field models with one index;
   real-world schemas have discriminated unions, lazy/recursive types,
   `.extend()` chains, deep nesting.

The dan-myles repro inlines 8 schemas per endpoint (discriminated unions,
lazy recursion, `z.record(union)`) and observed OOM at ~70 endpoints under
the *old* aggregate constraint. Under per-endpoint analysis its single-bundle
cost is more interesting than its endpoint count.

## Notable curiosity

`zodvex-mini` bundles ~25 KB **larger** than `zodvex` and shows a slightly
higher heap delta. Possible causes: mini entrypoint imports both runtimes,
conditions resolver isn't selecting the mini package, or seed
`import { z } from 'zod'` in models bleeds in even with the mini server.
Worth a closer look before drawing conclusions about mini's per-endpoint
advantage under the new backend.

## Harness

- `composeFlavor.ts` — produces a flavor-specific convex/ tree
- `bundle.ts` — esbuild per entry, matches `innerEsbuild` in
  `node_modules/convex/dist/esm/bundler/debugBundle.js`
- `measureChild.mjs` — loads one bundle in a fresh isolate, prints JSON
- `measureBundle.ts` — spawns the child, returns parsed metric
- `bench.ts` — orchestrates: compose → bundle → measure → distribution

Wall time at N=200 sample=40: ~1.3 s per flavor. No Convex deploy.

```bash
bun run bench.ts --all --count=200 --sample=40 --out=results/baseline-200.json
bun run bench.ts --flavor=zodvex --count=500 --keep
```

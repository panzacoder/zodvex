# Authoritative ceiling sweep (with deployment resets)

Single dated snapshot of where each flavor breaks on a fresh
deployment diff. Each test pushes an empty placeholder schema first
so the next deploy's `finish_push` transaction diffs `0 → N`
(additions only), not `M → N` (replacing prior state). This is the
correct methodology — without it, residual state from prior tests
artificially shrinks the diff and masks the true single-push ceiling.

**Important correction**: earlier results in
`post-consolidation-2026-05-13.md` and `cross-library-real-deploy-2026-05-13.md`
reported zodvex reaching N=2000 and convex-helpers/zod3 reaching N=2500.
Those were diff-stacking artifacts — running tests back-to-back on the
same deployment, each push only had to diff a few hundred functions
against the previous push's state. With the reset methodology used
here, the true single-deploy ceiling for all memory-OK flavors is much
lower.

## Two distinct ceilings

**OOM (memory)**: 64 MB-per-isolate cap on Convex's deploy analyzer.
The story this PR addresses. Pre-overhaul zodvex hit this at ~N=155;
post-overhaul, zodvex doesn't hit OOM at any tested N. Plain
convex-helpers + zod4 still hits OOM at N=500 (no in-library fix
path).

**TooManyReads (transaction read-set)**: 4,096 distinct read intervals
per transaction. Hit during `finish_push` for *any* flavor that gets
through OOM. The deploy commits the entire schema+function-handle diff
in a single backend transaction. At ~N=800 (4,000 functions, ~5,000
indexes), the diff exceeds the read-set cap on a fresh deployment.

This is Convex's hard backend constant
(`TRANSACTION_MAX_READ_SET_INTERVALS = 4096` in
`convex-backend/crates/common/src/knobs.rs`). Not zodvex-specific.

## What zodvex's overhaul actually buys

The improvement table at the right scale:

| Configuration | OOM ceiling (before fix) | OOM ceiling (after fix) |
|---|---:|---:|
| zodvex (default) | N≈155 | not OOM-bound |
| zodvex + slim models | N≈350 | not OOM-bound |
| zodvex/mini | N≈425 | not OOM-bound |
| zodvex/mini + slim | N≈700 | not OOM-bound |

Now memory-bound flavors are limited by Convex's transaction-level
TooManyReads at ~N=800 rather than by their per-isolate heap
footprint. zodvex has the same ceiling as plain Convex apps. The
remaining wall is **architectural to Convex**, not zodvex's domain.

## Why the regression target dropped to N=600

`regression.ts`'s previous default of N=800 was set against
diff-stacked deployment state. With clean resets — the correct
methodology — N=800 hits TooManyReads regardless of flavor.

N=600 was picked as the new target because:

1. **Comfortably below the TooManyReads wall**: 25% buffer below the
   N=750/800 boundary where the wall fires.
2. **Above ch/zod4's OOM at N=500**: the apples-to-apples comparison
   (zodvex passes, ch/zod4 OOMs) remains intact.
3. **Real-world-large**: 3,000 functions is more than most apps will
   ever have. Convex's published function-count limit is 8,192.
4. **No diff-stacking**: with the reset methodology, every regression
   run is a true single-deploy ceiling check.

## Methodology

- Each test pushes a placeholder schema (1 table, no functions) first
  via `resetDeployment()` in `realDeploy.ts`. Confirmed clean by
  observing the reset's own deploy outcome (always `ok`).
- After reset: compose at target N, run our local bench (heap proxy),
  push to Convex via `npx convex dev --once --typecheck=disable`,
  classify outcome.
- Tests halt at first failure for each flavor (no point pushing higher
  N once we've established the wall) via `--skip-after-failure`.
  Override with `--continue` to force every N regardless.

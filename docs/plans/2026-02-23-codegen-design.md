# Codegen Design

**Date:** 2026-02-23
**Status:** Approved
**Branch:** feat/codec-end-to-end

## Overview

zodvex ships a standalone CLI that generates two files into `convex/_zodvex/` by importing user modules at runtime and reading `__zodvexMeta` from exported functions and models. This enables client-side auto-decoding of query results and a single import point for all model schemas.

## Design Decisions

### Why codegen at all?

The client needs Zod schemas for args/returns to auto-decode query results (e.g., timestamps to Dates). These schemas are defined inline in server function definitions (`zq({ args: { email: z.string() }, ... })`). The client has no access to server code — codegen bridges this gap.

### Why not a Convex plugin?

Convex has no public plugin API for third-party code generation. The `_generated/` directory is exclusively managed by Convex's CLI, which actively deletes unknown files (see `doCodegen()` cleanup in `convex/src/cli/lib/codegen.ts`, lines 143-148). Convex Components (`defineApp`) provide data isolation, which is the opposite of what zodvex needs.

### Why not inside `_generated/`?

Convex's codegen runs a cleanup step that deletes anything in `_generated/` it didn't write. Even with careful ordering (zodvex running after Convex), there's a race condition during watch mode. `convex/_zodvex/` is safe from this cleanup while still inside the `convex/` directory for bundling.

### Why runtime execution instead of AST parsing?

AST parsing is fragile — breaks on aliased imports, spread operators, ternaries, re-exports, and any indirection. Runtime execution via `__zodvexMeta` is stable regardless of how users compose their code. The CLI imports modules and reads properties from objects. No pattern matching.

### Why a standalone CLI instead of runtime-only?

Pure runtime (no codegen) was explored but rejected. The validator registry — mapping function references to Zod schemas — is the core deliverable. On the server, `__zodvexMeta` works at runtime. On the client, there's no way to access server function objects — `api.foo.bar` is just an opaque string path. The client needs a generated file that maps these paths to schemas.

### Why not "side effect of import"?

`convex dev` uses esbuild to bundle modules — it parses and transforms but does NOT execute them. Module-level side effects never fire during local bundling. Execution only happens on the sandboxed Convex server (no filesystem access). There is no local execution step to piggyback on.

## CLI Commands

zodvex ships a CLI binary via `package.json`:

```json
"bin": { "zodvex": "./dist/cli.js" }
```

### `zodvex generate`

One-shot codegen. Imports all convex/ modules, reads `__zodvexMeta`, writes generated files. For CI/deploy pipelines.

### `zodvex dev`

Watch mode. Runs initial generation, then watches convex/ for changes and regenerates. For local development alongside `convex dev`.

### `zodvex init`

Interactive setup command that integrates zodvex codegen into an existing Convex project:

1. **Detects `convex dev`** in `package.json` scripts
2. **Installs `concurrently`** as a devDependency if not already present
3. **Rewrites dev script**: `"dev": "bunx convex dev"` becomes `"dev": "concurrently \"zodvex dev\" \"convex dev\""`
4. **Adds codegen to deploy/build scripts**: prefixes with `zodvex generate &&`
5. **Configures .gitignore**: adds `convex/_zodvex/` (generated files)

Example transformation for hotpot:

```json
// Before
"dev": "bunx convex dev",

// After
"dev": "concurrently \"zodvex dev\" \"bunx convex dev\"",
"deploy": "zodvex generate && bunx convex deploy"
```

## Discovery: `__zodvexMeta`

### Function metadata

Builders (`zq`, `zm`, `za`, `ziq`, `zim`, `zia`) attach metadata to the function object they return:

```ts
fn.__zodvexMeta = {
  type: 'function',
  zodArgs: argsSchema,      // z.ZodObject or undefined
  zodReturns: returnsSchema  // z.ZodType or undefined
}
```

This happens automatically inside the builder — no user action required. The metadata is the source of truth because it's captured from the actual runtime values, not parsed from source text.

### Model metadata

`defineZodModel` attaches metadata to the model object:

```ts
model.__zodvexMeta = {
  type: 'model',
  tableName: 'users',
  schemas: { doc, insert, update, docArray }
}
```

### CLI discovery process

1. Resolve the `convex/` directory
2. Find all `.ts`/`.js` files (excluding `_generated/`, `_zodvex/`, `node_modules/`)
3. Import each module via Bun/tsx runtime
4. Iterate over exports, collect anything with `__zodvexMeta`
5. Build the registry and schema re-exports
6. Write generated files

## Generated Files

### `convex/_zodvex/schema.ts`

Convenience re-exports of all `defineZodModel` instances. Gives client code a single import point for all models:

```ts
// AUTO-GENERATED by zodvex — do not edit
// Run `zodvex generate` to regenerate

export { UserModel } from '../models/user'
export { PostModel } from '../models/post'
export { CommentModel } from '../models/comment'
```

### `convex/_zodvex/validators.ts`

Maps Convex function paths to their Zod arg/return schemas. Uses direct schema references (not serialized representations):

```ts
// AUTO-GENERATED by zodvex — do not edit
// Run `zodvex generate` to regenerate

import { z } from 'zod'
import { zx } from 'zodvex/core'
import { UserModel } from '../models/user'

export const zodvexRegistry = {
  'users:get': {
    args: z.object({ id: zx.id('users') }),
    returns: UserModel.schema.doc,
  },
  'users:create': {
    args: z.object({ name: z.string(), email: z.string() }),
    returns: zx.id('users'),
  },
  'users:list': {
    args: z.object({}),
    returns: UserModel.schema.docArray,
  },
} as const
```

The registry keys use Convex's function path format (`"module:exportName"`), matching the string paths that `api.module.exportName` resolves to at runtime.

## Client API: `zodvex/react`

### `useZodQuery`

Wraps Convex's `useQuery` with auto-decode via the registry:

```tsx
import { useZodQuery } from 'zodvex/react'
import { api } from '../convex/_generated/api'

function UserProfile({ id }: { id: string }) {
  const user = useZodQuery(api.users.get, { id })
  // user.createdAt is Date (auto-decoded from timestamp)
  // user.updatedAt is Date (auto-decoded from timestamp)
  return <div>Joined: {user?.createdAt.toLocaleDateString()}</div>
}
```

### `useZodMutation` / `useZodAction`

Same pattern for mutations and actions — auto-encode args, auto-decode returns:

```tsx
import { useZodMutation } from 'zodvex/react'

function CreateUser() {
  const create = useZodMutation(api.users.create)
  // Args are auto-encoded (Date → timestamp, etc.)
  return <button onClick={() => create({ name: 'Alice', joinedAt: new Date() })}>Create</button>
}
```

### Implementation sketch

```ts
import { useQuery as convexUseQuery } from 'convex/react'
import { zodvexRegistry } from '../convex/_zodvex/validators'
import { decodeValue } from 'zodvex/core'

export function useZodQuery(fnRef, ...args) {
  const raw = convexUseQuery(fnRef, ...args)
  const fnName = getFunctionName(fnRef) // reads Symbol.for("functionName")
  const entry = zodvexRegistry[fnName]
  if (raw == null || !entry?.returns) return raw
  return decodeValue(entry.returns, raw)
}
```

## Documentation Requirements

This feature requires thorough documentation because it introduces a new CLI tool and changes the developer's workflow:

1. **Quick start guide** — `zodvex init` walkthrough, from zero to working codegen
2. **How it works** — explain `__zodvexMeta`, runtime import, generated files
3. **Troubleshooting** — common issues (stale files, import errors, missing metadata)
4. **Migration guide** — for existing zodvex users upgrading to codegen
5. **API reference** — `useZodQuery`, `useZodMutation`, `useZodAction`, registry format
6. **Architecture** — why codegen (not runtime-only), why `_zodvex/` (not `_generated/`), why runtime execution (not AST)

## Interaction with Existing Architecture

### Server-side (no change needed)

`initZodvex` + builders + DB wrapping continue to work as-is. The codegen is additive — it doesn't change server-side behavior. `__zodvexMeta` is attached as a non-enumerable property that doesn't affect Convex's function registration.

### Client-side (new)

`zodvex/react` is a new export path. It depends on the generated `validators.ts` for the registry. Without codegen, the wrapper hooks are not available — users fall back to vanilla Convex hooks with manual decode.

### defineZodModel (enhanced)

Models gain `__zodvexMeta` for automatic discovery. The generated `schema.ts` re-exports them, giving clients a single import point. This is purely additive — models work identically without codegen.

## Open Questions

1. **Inline schema serialization** — For ad-hoc args like `{ email: z.string() }`, the generated file needs to reproduce the Zod expression. Direct `z.string()` calls are simple, but complex schemas (unions, transforms, codecs) may need careful handling. Scope: address during implementation.

2. **Watch mode debouncing** — How long should `zodvex dev` wait after a file change before regenerating? Needs to be long enough to avoid thrashing but short enough to feel instant.

3. **Error handling** — What happens when a module fails to import? The CLI should log the error and skip that module, not crash the watcher.

4. **Type generation** — Should `validators.ts` also emit `.d.ts` files for enhanced type inference in the wrapper hooks? Or can the types be inferred from the runtime registry object?

# zodvex 0.8.0-beta.0 — hotpot trial script

A structured trial for taking `zodvex@0.8.0-beta.0` into the hotpot project.
This beta is the codec-paths memory-ceiling overhaul (PR
[#80](https://github.com/panzacoder/zodvex/pull/80)). It is a **larger change
than a normal beta** — it bumps the Convex SDK, changes the generated
`_zodvex/` file set, and **changes how `ctx.db` decodes/encodes**. Trial it
deliberately.

Two risk classes, in priority order:

1. **Does it deploy?** (the win — codecs at pure-Convex parity instead of
   OOMing the per-entrypoint isolates).
2. **Does codec data still round-trip correctly?** (the danger — hotpot's
   `SensitiveField` encryption rides the `ctx.db` boundary, and a silent
   decode miss means ciphertext/wire data flowing into app code as if
   decoded). **Section 6 is the one that matters most for hotpot.**

---

## 0. Baseline — capture BEFORE touching anything

On the current (pre-beta) hotpot state:

- [ ] Record current `zodvex` and `convex` versions (`grep -E 'zodvex|"convex"' package.json`).
- [ ] **Does it currently deploy, and how close to the limit?** Run your normal `convex deploy` (to a dev/throwaway deployment) and save the output. If you've seen "JavaScript execution ran out of memory" or have been avoiding adding tables, note that — this beta's whole point is to clear it.
- [ ] Count: total tables, tables that use a codec, and specifically which tables carry `SensitiveField`. Note any models that use a zod **union** to hold a codec.
- [ ] Run the **full existing test suite green** on the current version. This is your correctness baseline for §6.
- [ ] Branch: `git checkout -b trial/zodvex-0.8.0-beta`.

---

## 1. Install

- [ ] `npm i zodvex@0.8.0-beta.0` (or your package manager's equivalent).
- [ ] Bump peers: `convex@^1.41.0`, `convex-helpers@^0.1.119`. **This is a Convex SDK bump (1.x → 1.41), not just zodvex** — skim the convex-js changelog 1.33→1.41 for anything that touches you (notably `db.get(table, id)` is now preferred but the old form still works; `transactionLimits` on `runQuery`/`runMutation`; `ctx.meta.*`).
- [ ] `npm install` / reinstall, confirm a single `convex` version resolves (no duplicate-version drift).

---

## 2. Regenerate + INSPECT — the most informative step

- [ ] Run `zodvex generate` (or `npx zodvex dev` once). **Read the output carefully.**
- [ ] **Note every `falls back to its full model` warning.** Each names a table whose codecs the minimal-descriptor format couldn't express (codec inside a non-discriminated union/record/tuple, or a codec with no importable standalone reference). A fallback table is *correct* but pays its full model graph in every endpoint bundle. For each fallback, decide if it's acceptable or worth refactoring (e.g. export the inline codec standalone, or make the union discriminated).
- [ ] **Open `convex/_zodvex/models/<table>.js` for your `SensitiveField` tables.** Confirm `SensitiveField` shows up as an `import { … } from '…'` (minimal descriptor) rather than a fallback. If `SensitiveField` is a standalone export or a branded factory it should resolve minimal; if it's defined inline inside a model file with no export, that table falls back. **This is the single most useful thing to check** — it tells you the encryption codec is being discovered correctly.
- [ ] Confirm the new artifacts exist: `_zodvex/tables.ts`, `_zodvex/models/` (+ `index.js`), `_zodvex/api.args.js`, `_zodvex/server.ts`, `_zodvex/convex.config.ts`.

---

## 3. Migrate the consumer shape

- [ ] `bun zodvex migrate ./convex --dry-run` first; review, then run for real.
- [ ] After migration, `convex/schema.ts` should be `defineSchema(tables)` importing `./_zodvex/tables`, and `convex/functions.ts` should be one import + one call: `initZodvex({...builders})` from `./_zodvex/server`.
- [ ] **Missing-registry footgun is gone** — the generated `server.ts` auto-wires the registry. But explicitly confirm your `functions.ts` uses the generated `_zodvex/server` `initZodvex`, not a hand-rolled `initZodvex(schema, server)` left over from before. (This is the exact misconfiguration the 0.7.5-beta trial hit.)

---

## 4. Typecheck

- [ ] `npx convex codegen && npx tsc --noEmit` (your usual typecheck).
- [ ] Watch for: anything referencing the old `_zodvex/server` value-exports, or code that imported the schema object expecting the full runtime schema (the `schema` re-export from `server.ts` is now **type-full but runtime-token-only** — it carries the tableMap + decoded-doc token at runtime, not the table definitions).

---

## 5. Deploy — THE CEILING TEST

- [ ] Deploy to a throwaway/dev deployment.
- [ ] **PASS = deploys clean, no `JavaScript execution ran out of memory`.** If you were near or over the ceiling on baseline (§0), record before/after — this is the headline result of the whole beta.
- [ ] Fire the runtime smoke: hit a query, a mutation, and (if you have one) an action that does `ctx.runQuery`/`runMutation` — confirm no "dynamic module import unsupported" or argument-validation errors at runtime.

---

## 6. Correctness battery — THE DANGER ZONE (do not skip)

> **Why this section is the priority.** In 0.8, the `ctx.db` tableMap is the
> generated per-table descriptors — **minimal loose schemas that decode/encode
> ONLY codec fields and pass everything else through untouched.** In 0.7.x the
> tableMap held the *full* model schemas, so `ctx.db` reads/writes ran a full
> zod parse. Two consequences to verify:

### 6a. Codec round-trips must be correct (the silent-miss check)

- [ ] **`SensitiveField` direct round-trip:** via `ctx.db`, insert a doc with an encrypted field, read it back in the same/another query, and assert it comes back **decrypted (runtime shape)**, not the wire/ciphertext shape. Do this for every table that carries `SensitiveField`.
- [ ] **`SensitiveField` RELATIONAL round-trip (the dangerous case):** from a function that does **not** directly query the SensitiveField table, follow a foreign key into it — `const row = await ctx.db.get(otherDoc.sensitiveTableId)` — and assert the encrypted field decrypted. This is the exact pattern that silently returned wire data under the rejected per-endpoint design; confirm it decodes under codec-paths.
- [ ] **`zx.date()` and any other codecs:** same insert→read assertion (number ↔ Date, etc.).
- [ ] **`patch()` with mixed fields:** patch a codec field + a plain field together; assert the codec field encoded correctly AND the plain field was written (the 0.8 `encodePartialDoc` fix changed patch from a *strip* object to *loose* — fields outside the schema shape now pass through instead of being silently dropped).

### 6b. SEMANTIC CHANGE — non-codec read/write behavior (most likely surprise)

`ctx.db` reads/writes no longer run a full model zod parse — only codec fields
transform; everything else passes through loose. **If any model relies on
zod behavior beyond codecs at the `ctx.db` boundary, it is gone in 0.8:**

- [ ] Audit models for `.default()`, `.transform()`, or `.refine()` on **non-codec** fields that you depended on being applied **on a `ctx.db` read or write** (not in function args — function-arg validation is unchanged). Example: a non-codec field with a zod default that you relied on being filled in when inserting a partial via `ctx.db.insert`. That default will not be applied now.
- [ ] Note: Convex's own deployed validators (from `tables.ts`) still enforce document **structure** at write — this change is about zod-specific coercion/defaults/refinements at the zodvex layer, not raw schema validation.
- [ ] If you find a real dependency here, flag it — it's the one behavior this design trades away, and we should decide whether to address it before stable.

### 6c. Rules / audit

- [ ] Run any `.withRules()` / `.audit()` tests. These construct from the wrapped reader/writer internals and should pass through the descriptor tableMap unchanged — confirm enforcement still fires AND decoded values still reach rule predicates.

### 6d. Scheduler + actions + transactionLimits

- [ ] `ctx.scheduler.runAfter`/`runAt` with codec args (e.g. a `Date`/`SensitiveField` arg): assert the scheduled function receives correctly-encoded args (these now route through the minimal `api.args.js`).
- [ ] Action `ctx.runQuery`/`runMutation` returning codec data: assert results decode.
- [ ] If you pass `transactionLimits` to `runQuery`/`runMutation`, confirm it's honored (0.8 fixes a bug where zodvex's wrapper silently dropped that options arg).

### 6e. Full suite

- [ ] Run hotpot's **entire** test suite (unit + integration + e2e) and diff against the §0 baseline. Any new failure — especially anything where a value comes back as a `number`/`string`/ciphertext where a decoded type was expected — is a decode-miss and a hard blocker.

---

## 7. CI staleness guard

- [ ] Add `npx zodvex generate --check` to CI (or a pre-commit hook). Confirm it passes on the freshly-committed `_zodvex/` output, then tamper a model (add a `zx.date()` field, don't regenerate) and confirm `--check` exits non-zero and names the stale files. This is what stops "edited a model, forgot to regenerate" from silently breaking decode in a future PR.

---

## 8. Red flags to watch throughout

- A value returning as `number` / `string` / **ciphertext** where a `Date` / decrypted runtime value was expected → **decode miss, stop and report.**
- A descriptor fallback on a hot/large table → not wrong, but that table re-pays its model graph per endpoint; if it's pushing you back toward the ceiling, refactor the codec to be standalone/branded or the union to be discriminated.
- `patch()` writing a different field set than 0.7.x did (strip → loose change).
- Type errors around the `schema` re-export (now runtime-token-only).

---

## 9. Rollback

It's a beta on a trial branch — reverting is clean:

- [ ] `npm i zodvex@<previous-version>`, restore the prior `convex` / `convex-helpers` versions.
- [ ] `git checkout main -- convex/_zodvex convex/schema.ts convex/functions.ts` (and any migrated files), or `git checkout main` the branch entirely.
- [ ] Redeploy to confirm the revert is clean.

## Report back

When done, the useful signal for the zodvex side:

1. **Deploy:** did it deploy, and before/after memory headroom if you had a ceiling.
2. **Fallbacks:** which tables fell back to full models, and why (esp. any `SensitiveField` table).
3. **§6a correctness:** any decode miss, especially on relational `SensitiveField` reads.
4. **§6b semantic change:** did the loss of full-doc zod parse on `ctx.db` reads/writes affect anything real.
5. **§6c–d:** rules/audit, scheduler, actions, transactionLimits — any regressions.

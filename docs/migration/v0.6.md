# Migrating to zodvex v0.6

v0.6 closes the codec gap: previously, `zx.date()` and custom codecs defined *what* transformations should happen, but the actual encode/decode only ran at function argument boundaries — not when reading from or writing to the database. This meant codec fields like dates came back as raw timestamps from `ctx.db.get()`, forcing manual conversion. `initZodvex` fixes this by wrapping `ctx.db` with automatic codec handling at all boundaries. No codegen or CLI is required — just `initZodvex`.

## Quick Start

Run the automated codemod to handle mechanical renames:

```bash
npx zodvex migrate ./convex
# or
bunx zodvex migrate ./convex
```

The codemod handles:

- **Identifier renames**: `CodecDatabaseReader` -> `ZodvexDatabaseReader`, `CodecDatabaseWriter` -> `ZodvexDatabaseWriter`, `CodecQueryChain` -> `ZodvexQueryChain`, `CodecRules` -> `ZodvexRules`, etc.
- **`zid()` -> `zx.id()`**: Standalone `zid(` calls become `zx.id(`, and import specifiers are updated automatically.
- **Import specifier cleanup**: `zid` is removed from `zodvex` imports and `zx` is added if not already present.

After running the codemod, check your terminal output for **remaining deprecation warnings** -- these are API changes that require manual migration (covered below).

Use `--dry-run` to preview changes without writing files:

```bash
npx zodvex migrate ./convex --dry-run
```

## zodTable -> defineZodModel + defineZodSchema

`zodTable` combined model definition and Convex table creation in one call. v0.6 splits these concerns: `defineZodModel` defines the Zod schema (client-safe), and `defineZodSchema` creates the Convex schema (server-only).

### Before

```ts
// convex/schema.ts
import { defineSchema } from 'convex/server'
import { zodTable } from 'zodvex/server'
import { z } from 'zod'

const Users = zodTable('users', {
  name: z.string(),
  email: z.string().email(),
})

export default defineSchema({ users: Users.table })
```

### After

```ts
// convex/models.ts (client-safe -- can import in React)
import { defineZodModel } from 'zodvex/core'
import { z } from 'zod'

export const Users = defineZodModel('users', {
  name: z.string(),
  email: z.string().email(),
})

// convex/schema.ts (server-only)
import { defineZodSchema } from 'zodvex/server'
import { Users } from './models'

export default defineZodSchema({ users: Users })
```

### Key differences

| v0.5 (`zodTable`)                | v0.6 (`defineZodModel`)                          |
| -------------------------------- | ------------------------------------------------ |
| Server-only                      | Client-safe (importable from React)               |
| `.shape` for field access        | `.fields` for field access                        |
| `.zDoc` / top-level `.docArray`  | Removed -- use `.schema.doc` / `.schema.docArray` |
| Indexes via `defineTable` chain  | Chainable `.index()`, `.searchIndex()`, `.vectorIndex()` |

The `.schema` sub-object (`.schema.doc`, `.schema.docArray`, `.schema.base`, `.schema.insert`, `.schema.update`, `.schema.paginatedDoc`) is unchanged.

### Indexes

```ts
export const Users = defineZodModel('users', {
  name: z.string(),
  email: z.string().email(),
  teamId: zx.id('teams'),
})
  .index('byEmail', ['email'])
  .index('byTeam', ['teamId'])
  .searchIndex('search', { searchField: 'name', filterFields: ['teamId'] })
```

## Builder -> initZodvex

Individual builder functions (`zQueryBuilder`, `zMutationBuilder`, etc.) are replaced by a single `initZodvex` call that returns all builders pre-configured with your schema.

### Before

```ts
// convex/queries.ts
import { query } from './_generated/server'
import { zQueryBuilder } from 'zodvex/server'

const zq = zQueryBuilder(query)

export const getUser = zq({
  args: { id: z.string() },
  handler: async (ctx, { id }) => ctx.db.get(id),
})
```

### After

```ts
// convex/zodvex.ts (one-time setup)
import { initZodvex } from 'zodvex/server'
import {
  query, mutation, action,
  internalQuery, internalMutation, internalAction,
} from './_generated/server'
import schema from './schema'

export const { zq, zm, za, ziq, zim, zia } = initZodvex(schema, {
  query, mutation, action,
  internalQuery, internalMutation, internalAction,
})

// convex/queries.ts
import { zq } from './zodvex'

export const getUser = zq({
  args: { id: z.string() },
  handler: async (ctx, { id }) => ctx.db.get(id),
})
```

`initZodvex` returns builders for all six Convex function types:

| Builder | Function type      |
| ------- | ------------------ |
| `zq`    | public query       |
| `zm`    | public mutation    |
| `za`    | public action      |
| `ziq`   | internal query     |
| `zim`   | internal mutation   |
| `zia`   | internal action    |

By default, `zq` and `zm` (and their internal variants) wrap `ctx.db` with automatic codec encode/decode. Pass `{ wrapDb: false }` as the third argument to opt out.

### Custom builders

If you were using `zCustomQueryBuilder` with a custom context, use `.withContext()` instead:

**Before:**

```ts
import { zCustomQueryBuilder } from 'zodvex/server'
import { customCtx } from 'convex-helpers/server/customFunctions'

const zq = zCustomQueryBuilder(query, customCtx(async (ctx) => {
  const user = await getAuthUser(ctx)
  return { ctx: { user } }
}))
```

**After:**

```ts
import { customCtx } from 'zodvex/server'

const authedQuery = zq.withContext(customCtx(async (ctx) => {
  const user = await getAuthUser(ctx)
  return { user }
}))
```

The same `.withContext()` pattern works for `zm`, `za`, and all internal variants.

## convexCodec -> decodeDoc/encodeDoc

The `convexCodec` factory is replaced by standalone `decodeDoc` and `encodeDoc` functions.

### Before

```ts
import { convexCodec } from 'zodvex'

const codec = convexCodec(UserSchema)
const encoded = codec.encode(user)
const decoded = codec.decode(doc)
```

### After

```ts
import { encodeDoc, decodeDoc } from 'zodvex/core'

const encoded = encodeDoc(UserSchema, user)
const decoded = decodeDoc(UserSchema, doc)
```

**If you are using `initZodvex`**, you likely don't need `decodeDoc`/`encodeDoc` at all. The DB wrappers on `ctx.db` handle encode/decode automatically for all reads and writes. Direct use is only needed for escape-hatch scenarios (e.g., custom DB layers or manual Convex client calls).

## mapDateFieldToNumber -> zx.date()

Manual date mapping is replaced by a declarative codec.

### Before

```ts
import { mapDateFieldToNumber } from 'zodvex'

const schema = z.object({
  createdAt: z.date(),
})
// + manual mapDateFieldToNumber calls at encode/decode boundaries
```

### After

```ts
import { zx } from 'zodvex/core'

const schema = z.object({
  createdAt: zx.date(),
})
// No manual mapping needed -- zx.date() handles Date <-> timestamp automatically
```

`zx.date()` is a Zod codec that:
- **Stores** a number (Unix timestamp in ms) in Convex
- **Parses** to a `Date` object at runtime
- **Encodes** back to a number on writes

It works with `.optional()`, `.nullable()`, and inside arrays/unions with no extra configuration.

## Removed Internals

These symbols were public in pre-0.6 but are now internal. If your code imports any of them, you will need to refactor:

| Removed symbol | Replacement |
| --- | --- |
| `customFnBuilder` | `zCustomQuery` / `zCustomMutation` / `zCustomAction` |
| `registryHelpers` | `zx.id()` -- zodvex handles metadata internally |
| `makeUnion` | `zodToConvex` handles unions internally |
| `pick` | Use your own pick utility or lodash |
| `formatZodIssues` | zodvex handles validation internally |
| `handleZodValidationError` | zodvex handles validation internally |
| `validateReturns` | zodvex handles validation internally |
| `assertNoNativeZodDate` | zodvex checks this automatically at function registration time |
| `isZodUnion` | `schema instanceof z.ZodUnion` (standard Zod API) |
| `getUnionOptions` | `schema.options` (standard Zod API) |
| `assertUnionOptions` | `schema.options` (standard Zod API) |
| `createUnionFromOptions` | `z.union(options)` (standard Zod API) |
| `attachMeta` / `readMeta` | Internal to zodvex's codegen system |
